To understand C++, we first need to understand programming. Programming is basically a way how to express some functionality in a specific human-“readable” form, which differs from language to language. We then want this written functionality — called source code — to be performed on a computer system. More precisely, performed by its CPU (we will not cover here special cases such as GPU programming).

CPUs do not understand source code. They understand only a single “language” — the machine code. This inevitably leads to the conclusion that to execute the desired functionality, its source code must be first translated into machine code. This is a role of implementations of programming languages with their specific tools (compilers, interpreters, linkers,…​).

For example, consider the following source code of a C++ function:

int absmax(int a, int b)
{
  int abs_a = std::abs(a);
  int abs_b = std::abs(b);
  return std::max(abs_a, abs_b);
}

Its functionality is taking two integer numbers and returning a higher of their absolute values. When I compiled this source code with GCC on an x86_64 system, I obtained the following machine code, printed out in hexadecimal format:

89 fa 89 f0 c1 fa
1f 31 d7 29 d7 99
31 d0 29 d0 39 c7
0f 4d c7 c3

These are the numbers (called opcodes) my CPU understands. We can understand them as well by studying the Intel Instruction Set Reference or similar sources. For instance, we can read there that the last c3 opcode corresponds to the ret instruction.

However, let’s get back to programming languages. How do they differ? Among others — such as that they use a different source code form/syntax — each language provides a different set of abstraction mechanisms. An abstraction mechanism is a programming language entity/concept that allows us to express some desired functionality. Exemplary abstraction mechanisms are variables, functions (with parameters and return values), loop constructs, structures, classes, inheritance, polymorphism, templates, reflection, and many more.

The important aspect of abstraction mechanisms is that they do not exist at a machine code level, that is, they do not exist at runtime. If we decoded the above-presented machine code, we would realize that it contained only the instructions for register manipulations and transferring control back to a function caller.

In the next post, we will look at how we can classify programming languages according to the abstraction mechanisms they provide.
